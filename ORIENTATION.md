# Coon Community Builder — Codebase Orientation

> Generated 2026-04-16. Re-generate when architecture changes significantly.

---

## 1. Architecture Map

Requests flow through **Clerk middleware** (`src/middleware.ts`) which protects all non-public routes via `clerkMiddleware` + `createRouteMatcher`. **API route handlers** (`src/app/api/`) are the entry point for all business logic: they call `await auth()` from `@clerk/nextjs/server` first, then parse the request body with Zod (inline for simple schemas, imported from `src/lib/validations/` for complex ones), then resolve services from the **DI container** (`src/lib/core/di/container.ts` — `getContainer()` singleton). **Services** (`src/lib/core/services/`) orchestrate between **repository interfaces** (`src/lib/core/repositories/interfaces.ts`) and **agent functions** (`src/lib/agents/`). Repositories are Drizzle ORM implementations that talk to **Neon Postgres** via `src/lib/db/index.ts`. Agent functions call the Claude API via the shared `anthropic` client from `src/lib/claude.ts`, validate output with Zod, and return `{ result, modelUsed, tokensUsed }`. The **orchestration layer** (`src/lib/orchestration/`) provides queue, circuit breaker, retry, and cache primitives — however these are currently **not wired into services**; services use `withRetry()` from `src/lib/agents/utils.ts` directly. On the frontend, **React Query hooks** (`src/hooks/`) wrap fetch calls to API routes, and **components** (`src/components/`) consume those hooks.

## 2. Database Conventions

- **Schema location**: Single file `src/lib/db/schema.ts` (387 lines), sectioned by domain with comment headers.
- **Connection**: `src/lib/db/index.ts` — Neon serverless HTTP driver via `drizzle-orm/neon-http`.
- **Table naming**: `snake_case` in SQL (e.g., `campaign_content`, `agent_runs`). Drizzle column definitions use `camelCase` JS names mapped to `snake_case` SQL names.
- **Primary keys**: `uuid("id").defaultRandom().primaryKey()` on every table.
- **User references**: `text("user_id").notNull().references(() => users.id, { onDelete: "cascade" })`. The `users` table PK is a Clerk user ID (`text`, not UUID).
- **Timestamps**: `timestamp("created_at").defaultNow()` and `timestamp("updated_at").defaultNow()`. There is **no database trigger** for `updatedAt` — services set `updatedAt: new Date()` manually in UPDATE calls.
- **Enums**: No `pgEnum` — all status fields use `text("status")` with TypeScript string literal unions in `src/types/index.ts`.
- **JSONB**: Used for AI-generated data (`profileData`, `strategyData`, `calendarData`, `contentData`, `frequencyConfig`, `metadata`, `analyticsData`, `cohesionResult`). Cast on read with TypeScript types from `@/types`.
- **Arrays**: `text("column").array()` for `hashtags`, `selectedPlatforms`, `completedPlatforms`, `scopes`.
- **Relations**: Defined alongside tables using Drizzle's `relations()` API with `one()` and `many()`.
- **Indexes**: Explicit on `campaigns(userId, status)`, `campaignContent(campaignId)`, `campaignContent(campaignId, status)`, `campaignContent(scheduledFor)`.
- **Migrations**: `src/lib/db/migrations/` with 4 files. Naming is mixed: first is manual (`0000_add-campaign-tables.sql`), rest are auto-generated by Drizzle Kit (`0001_dusty_tarantula.sql`, etc.). Config in `drizzle.config.ts`.

## 3. Repository Conventions

- **Pattern**: Interface-first. All contracts are in `src/lib/core/repositories/interfaces.ts`. Implementations are `drizzle-*.ts` files in the same directory.
- **Classes**: Repositories are classes instantiated with `database: typeof db` in the DI container constructor.
- **Standard methods**: `findById(id, userId)`, `findByUserId(userId)`, `create(params)`, `save(entity)`, `delete(id)`. No generic CRUD — each repo has domain-specific methods (e.g., `updatePlan()`, `updateStrategy()`, `updateCalendar()`, `bulkUpdateApprovalStatus()`).
- **Row mapping**: Each repo has a private `toDomain(row)` method that maps Drizzle rows to domain entity classes.
- **Filters**: Always filter by `userId` for tenant isolation. Use `and()`, `eq()` from `drizzle-orm`. Single-row fetches use destructuring: `const [item] = await db.select()...limit(1)`.
- **Returns**: INSERT and UPDATE always use `.returning()` and map through `toDomain()`.
- **Pagination**: Not implemented — all list methods return full result sets ordered by `desc(createdAt)`.
- **DI registration**: All 8 repos instantiated in `Container` constructor in `src/lib/core/di/container.ts`.

## 4. Service Conventions

- **Structure**: Classes with constructor injection of repository interfaces + agent function references + optional `PluginRunner`. Instantiated in the DI container.
- **Services**: `CampaignService` (1,298 lines), `AudienceService` (140 lines), `EnrichmentService` (226 lines), `PublishService`, `AnalyticsService`.
- **Error handling**: Throw `ServiceError` (defined in `src/lib/core/services/audience-service.ts`, re-exported from `src/lib/core/services/index.ts`). Error codes: `NOT_FOUND`, `NO_PROFILE`, `NO_QUIZ_RESPONSE`, `INVALID_STATE`, `AGENT_FAILED`.
- **Agent calls**: Services call injected agent functions, track `startTime = Date.now()`, compute `durationMs`, and call `agentRunRepo.log()` with success/failure status.
- **Token tracking**: Services accumulate `totalTokensUsed` from agent results and update campaign records.
- **Batch processing**: `BATCH_SIZE = 3` for concurrent content generation. Uses `Promise.allSettled` so individual failures don't block others.
- **State machine**: Campaign status transitions: `draft` -> `strategy_complete` -> `calendar_complete` -> `generating_content` -> `complete`. Services enforce valid transitions.
- **Plugins**: `PluginRunner` with `beforeExecution`/`afterExecution`/`onError` hooks. `TokenTrackingPlugin` and `DurationTrackingPlugin` are registered.

## 5. Agent Conventions

- **Location**: `src/lib/agents/{agent-name}.ts`. Platform-specific agents use subdirectories with `index.ts` + per-platform files.
- **Function signature**: `async function myAgent(input: InputType): Promise<{ result: OutputType; modelUsed: string; tokensUsed: number }>`.
- **Model**: `const MODEL = "claude-sonnet-4-20250514"` defined as file-level constant.
- **Prompt construction**: Separate `buildPrompt(input)` function returns a string. System prompt assigns role + "Respond with valid JSON only. No markdown, no explanation." User prompt has `##` sections with interpolated data and an exact JSON schema with field descriptions.
- **API call**: `anthropic.messages.create({ model: MODEL, max_tokens: 4096, system, messages: [{ role: "user", content: prompt }] })`.
- **JSON extraction**: `extractJSON(text)` from `src/lib/agents/utils.ts` — tries full parse, then markdown code blocks, then bracket-matching.
- **Validation**: Output parsed with Zod schema via `.parse()`. Schemas live in `src/lib/validations/`.
- **Retry**: Wrapped in `withRetry()` from `src/lib/agents/utils.ts` — exponential backoff, default 1 retry.
- **Logging**: `logAgentRun()` from `src/lib/agents/utils.ts` — called from API routes or services, NOT from agent functions themselves.
- **Token calculation**: `response.usage.input_tokens + response.usage.output_tokens`.
- **Existing agents**: `analyzeAudience`, `generateCampaignStrategy`, `generateCampaignCalendar`, `generateCampaignPlan`, `generateContentPiece`, `generatePlatformBatch`/`getNextBatch` (campaign-content), `checkCampaignCohesion`, `generateAnalyticsInsights`, `enrichContentWithMedia`/`isVisualPlatform`, `scoreContent`, `optimizeContent`.
- **Platform instructions**: `PLATFORM_INSTRUCTIONS` map in `content-piece-generator.ts` has detailed rules for all 11 platforms (reddit, instagram, tiktok, threads, youtube, twitter, blog, linkedin, pinterest, discord, email).
- **Tests**: Mock `@/lib/claude` (the anthropic client). Test: valid output, Zod validation catches malformed output, retry on API failure, correct model/prompt. Fixtures at top of test file or in `__fixtures__/`.

## 6. API Route Conventions

- **Auth (always first)**:
  ```typescript
  const { userId } = await auth();
  if (!userId)
    return NextResponse.json(
      { data: null, error: { message: "Unauthorized", code: "UNAUTHORIZED" } },
      { status: 401 }
    );
  ```
- **Service resolution**: `const { campaignService } = getContainer();`
- **Validation**: `safeParse()` for dual-path logic (creator vs legacy in campaign route), `.parse()` for single-path. Complex schemas imported from `src/lib/validations/`, inline `z.object()` for simple ones.
- **Response envelope**: `NextResponse.json({ data: result, error: null })` for success. `NextResponse.json({ data: null, error: { message, code } }, { status })` for errors.
- **Error mapping**: `ServiceError` caught and mapped — `NO_PROFILE`/`NO_QUIZ_RESPONSE` -> 400, `NOT_FOUND` -> 404, `AGENT_FAILED` -> 500.
- **Dynamic params**: `const { id: campaignId } = await params` (Next.js 15 async params pattern).
- **Long-running routes**: `export const maxDuration = 120` (or 300 for generation). Background execution via `after()` from `next/server` — returns immediately with status, runs work in background.
- **Outer try/catch**: Wraps entire handler, returns `INTERNAL_ERROR` on unexpected errors.

## 7. Hook Conventions

- **Location**: `src/hooks/use-{resource}.ts`. One file per resource, all hooks for that resource in the same file.
- **Naming**: `useCampaignList()`, `useCampaign(id)`, `useCreateCampaign()`, `useUpdateCampaign()`, `useDeleteCampaign()`, `useGenerateFullCampaign(campaignId)`.
- **Query keys**: List = `["campaigns"]`, single = `["campaign", id]`, nested = `["campaign", id, "content"]`, `["campaign", id, "generation-status"]`.
- **Query pattern**: `useQuery({ queryKey, queryFn, enabled })`. `queryFn` fetches, checks `!res.ok`, then checks `json.error`, throws on either.
- **Conditional queries**: `enabled: !!id` for single-item queries.
- **Polling**: `refetchInterval: enabled ? 2000 : false` for generation status.
- **Mutation pattern**: `useMutation({ mutationFn, onSuccess })`. `onSuccess` calls `queryClient.invalidateQueries({ queryKey })`.
- **Error handling**: All hooks throw `new Error(json.error.message)` — React Query surfaces via `.error`. No try/catch in queryFn/mutationFn.
- **No optimistic updates**: Current hooks invalidate on success, no optimistic UI.
- **No `"use client"` directive** in hook files — the consuming component provides it.
- **Provider**: `src/components/providers.tsx` — `QueryClientProvider` with `staleTime: 60_000`, `retry: 3`.

## 8. Component Conventions

- **File pattern**: `src/components/{feature}/{component-name}.tsx`. Named exports only, PascalCase.
- **Props**: Interface named `{ComponentName}Props`, destructured in function signature.
- **Client directive**: `"use client"` only when using hooks, event handlers, or browser APIs.
- **Styling**: Tailwind utility classes only. `cn()` from `@/lib/utils` for conditional classes. No CSS modules or styled-components.
- **shadcn/ui components available** (22 total): badge, button, card, checkbox, dialog, input, label, progress, radio-group, scroll-area, select, separator, sheet, skeleton, sonner (toast), tabs, textarea. Located in `src/components/ui/` — never modify these.
- **Forms**: React Hook Form + `zodResolver()`. `register()` for basic inputs, `Controller` for custom components, `watch()` for reactive state.
- **Drag-and-drop**: dnd-kit with `useSortable()`, `DndContext`, `DragOverlay`, closestCorners collision detection. Used in kanban board.
- **Loading states**: `isPending` from mutations for button disabled state. `sonner` toast for notifications.

## 9. Testing Conventions

- **Framework**: Vitest + React Testing Library + MSW.
- **File pattern**: `src/{path}/__tests__/{source-file}.test.ts` — co-located with source.
- **20 test files** exist covering: agents (3), hooks (2), validations (3), API routes (2), domain models (3), services (2), orchestration (4), plugins (1).
- **Agent tests**: Mock `@/lib/claude` anthropic client. Verify: correct model/prompt, JSON extraction, Zod validation catches malformed output, retry on failure, token calculation.
- **Service tests**: Mock all repositories and agents via `vi.fn()`. Type: `MockRepo<T> = { [K in keyof T]: ReturnType<typeof vi.fn> }`. Verify orchestration, error handling, agent run logging.
- **Hook tests**: Mock `fetch` globally via `vi.stubGlobal("fetch", mockFetch)`. Create wrapper with `QueryClient` (`retry: false`). Use `renderHook()` + `waitFor()`.
- **API route tests**: Mock `@clerk/nextjs/server` auth and `@/lib/db`. Must test 401 path.
- **Validation tests**: Use `.safeParse()` — test valid passes, invalid fails with expected errors.
- **No snapshot tests** — forbidden by standard.
- **No MSW handler files** found — tests mock fetch directly rather than using MSW server setup.

## 11. Gotchas & Non-Obvious Rules

1. **campaign-service.ts is 1,298 lines** — 4x the 300-line standard. Do not add more to it; extract into focused services when touching this file.

2. **types/index.ts is 607 lines** — 2x the 300-line standard. The standard mandates a single file, but it already needs sectioning discipline.

3. **`ServiceError` lives in `audience-service.ts`**, not in a shared module. Import it from `@/lib/core/services` (barrel re-export), not from the audience service directly.

4. **Orchestration layer is unused by services.** The `src/lib/orchestration/` module (queue, circuit breaker, pipeline, cache) is fully built and tested but no service imports it. All agent retry is handled by `withRetry()` in `src/lib/agents/utils.ts` (1 retry, exponential backoff). Don't build new orchestration primitives — wire the existing ones.

5. **`updatedAt` has no DB trigger.** Services must set `updatedAt: new Date()` explicitly in every UPDATE call. Forgetting this leaves stale timestamps.

6. **Next.js 15 async params.** Dynamic route params are `Promise<{ id: string }>` and must be awaited: `const { id } = await params`.

7. **Dual create paths in campaign POST.** The `POST /api/campaign` route supports both a `campaignCreatorSchema` (new flow creating a draft) and a `legacyCreateSchema` (old flow with just `selectedPlatforms` that immediately generates strategy). Check which flow you need.

8. **`after()` for background work.** The generate route uses `import { after } from "next/server"` to run `generateFullCampaign` in the background after returning a 200. This is a Next.js 15 feature — do not use `waitUntil` or manual background patterns.

9. **Content approval auto-sets on schedule.** `updateSchedule()` and `bulkUpdateSchedule()` in the content repo automatically set `approvalStatus: "approved"` — scheduling implies approval.

10. **`computeContentHash()` for cohesion caching.** The cohesion checker uses a SHA256 hash of content bodies to skip re-analysis if content hasn't changed. The hash is stored on the campaign record.

11. **Agent confidence scores stored as integers.** `aiConfidenceScore` is stored as `integer` (0-100), but agents return `confidenceScore` as a float (0-1). The repo does `Math.round(confidenceScore * 100)` in `updateContentPiece()`.

12. **No `"use client"` in hook files.** The standard says hooks don't have the directive — but `use-audience-profile.ts` DOES have `"use client"`. This inconsistency means you should add it if your hook file is consumed by a client component that doesn't already have it.

13. **`users.id` is `text`, not `uuid`.** The `users` table PK is the Clerk user ID (a string like `user_2abc...`), not a UUID. All `userId` foreign keys are `text`.

14. **Migration naming is inconsistent.** First migration is manually named (`0000_add-campaign-tables.sql`), rest are auto-generated by Drizzle Kit. Use `npx drizzle-kit generate` for new migrations.
